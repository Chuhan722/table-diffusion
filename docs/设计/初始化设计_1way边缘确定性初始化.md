# 初始化设计：按 1-way 边缘确定性初始化

> 归档日期：2026-07-24
> 状态：设计已定稿，尚未实现（遵守 CLAUDE.md：未经允许不实现后续阶段）
> 适用范围：扩散演化生成器的初始表 S_0 生成

## 1. 背景与动机

现状：`generator.init_synthetic_table` 生成的 S_0 是**纯随机**表——每格从 schema
合法域均匀抽样。随机起点连"每个属性自己的分布"都不对，演化的前若干轮被迫先去修
单变量分布，浪费在最基础的一维信息上。

目标：**用 1-way 边缘测量初始化 S_0**，让每个属性自己的分布一开始就贴合目标，
把演化的力气集中到属性间的高阶关系（如 3-way）上。

这一思路与 `docs/AIM_初始化机制详细说明.pdf` 的核心哲学一致：
> 先学习"便宜、稳定、几乎肯定需要"的单变量统计，再把昂贵的预算/工作量留给
> 属性间相关性。

区别：AIM 用 Private-PGM 拟合图模型再采样；我们走扩散演化路线，采用**确定性按配额
填充**（做法 B），不引入 PGM。两者结果语义相同：单变量分布对齐、属性间联合随机，
相关性交给后续机制去学。

## 2. 已确定的设计决策（与用户逐条确认）

1. **填充方式 = 做法 B（确定性按配额填充）**，不是随机采样。每属性按其 1-way
   边缘目标计数直接分配记录数，除取整误差外精确匹配目标。
2. **各属性独立生成**：每列按自己的边缘填，不看其他列；属性间联合关系交给后续
   3-way 演化去修。
3. **1-way 目标 = 带噪测量查询**，现阶段噪声设为 0，所以初始化直接对上无噪真实边缘；
   σ/κ 噪声接口后续 DP 阶段再用。
4. **边缘来源 = 单独维护一份初始化专用 marginals 文件**（与演化用的 measured 查询解耦）。
5. **数值属性用分箱（做法 1）**，还原数值的步骤**提前到初始化时**完成（我们没有
   "合成后还原"这一步，初始化时就直接生成真实数值）。
6. **保留 `INIT_METHOD` 开关**，可一键切回纯随机做对照实验，后续不需要再删。

## 3. 通用分箱规则：workload 诱导分箱

> 核心诉求：不是为某个数据集量身定制，而是换数据/换查询零改代码的通用方法。

规则一句话：**对每个数值属性，扫描全部查询里作用在它身上的条件，收集所有判定
边界（切点），排序去重，就是箱边界。**

### 3.1 为什么这是对的通用规则

因为整套方法是在**拟合一组测量查询**。初始化的目的是把 1-way 结构一次到位。
那么"该分多细"就由**查询能区分的粒度**决定：

- 比查询更细的箱 = 浪费：查询分辨不出同一区间内的差异（DP 阶段箱越细噪声占比越大）；
- 比查询更粗的箱 = 有害：一个箱横跨某查询阈值，这条查询在初始化就对不上；
- **正好等于查询切点** = 最粗但仍能精确匹配每条 1-way 查询的分箱，是最优粒度。

这与 AIM 文档同一哲学：分箱由 workload 决定，不靠盯着私有数据选。差异是 AIM 把
箱号当无序离散，我们保留数值顺序、还原成真实值（因为我们有 `>=`/`between` 和
有序距离 `distance.py`）。

### 3.2 统一处理，不为数值属性开特例

- **类别属性**：每个合法值天然是一个"箱"（nltcs 的 0/1 就是两个箱），是完整 1-way 直方图。
- **数值属性**：用查询切点分箱。
- **两者最终格式一致**：属性 → [(区间/值, 计数)]。
- **无查询覆盖的属性**：1-way 目标无从测量 → 运行时回退随机。

### 3.3 边界永远铺满整个域

造箱时总会并入 schema 域端点 `[dmin, dmax+1]`。因此无论查询给多少切点，箱永远
铺满整个域、计数之和恒为 N。这保证数值属性的 1-way 也是完整划分。

## 4. 算子白名单与回退（沿用 vectorized_eval 既定模式）

造箱本质是把每个查询的"判定边界"变成切点。不同算子切点位置不同，未处理的算子会
让箱横跨其阈值。沿用项目已有的"算子白名单 + 回退"模式（见 `vectorized_eval`：
白名单 `{==,>=,between}`，白名单外自动回退慢路径）。

### 4.1 数值算子切点（闭箱、整数域）

| 算子 | 切点 | 说明 |
|------|------|------|
| `>= t` | `t` | |
| `> t` | `t+1` | 等价 `>= t+1` |
| `<= t` | `t+1` | |
| `< t` | `t` | |
| `between [lo,hi]` 闭 | `lo`、`hi+1` | 端点含义严格对齐评价器 |
| `== v`（数值） | `v`、`v+1` | 退化箱 `[v,v]` |
| `!= v` | `v`、`v+1` | `[v,v]` 单独成箱 |

**端点语义**（开/闭、含不含端点）决定切点是 `t` 还是 `t+1`，差一位就错一位；
必须与 `queries.eval_condition` 的判定逻辑逐一对齐，并由单测钉死。

### 4.2 白名单外算子的行为

- **可预测的降级，不是脆断**：造箱**静默跳过该查询**（不贡献切点）+ 打印提醒。
- 后果不是报错或结果错误，而是**这条查询的 1-way 信息没烤进初始化**，起点差一点，
  交给演化去修。评价阶段该查询照样算对（走 `eval_condition`）。
- 一句话：新算子最坏情况是"初始化没帮上这条查询的忙"，不会导致结果错误——
  初始化只是给演化更好的起点，不是最终答案，演化仍以真实 target 为准。

## 5. 落到具体数值：箱内确定性均摊

- 某箱 `[lo,hi]`、配额 `count` 条 → 把 `count` 条**等距铺在 `lo..hi` 整数上**
  （如 `lo + i % (hi-lo+1)`），确定性、无随机。
- 对比"全填箱中点"：那样整列只有少数几个值，分布形状很假，且 `distance.py` 把
  age 当有序数值时，全挤中点会让距离结构失真。均摊形状更自然。
- **关键**：因边界对齐查询切点，箱内怎么铺都不改查询计数（配额仍全落在同一区间内），
  所以均摊只改善分布形状、不破坏要对齐的 1-way 目标。纯赚。
- 类别属性无此问题（箱就是单值）。
- 这是**确定性等距分配**，不是随机采样，与做法 B 一致，不引入新种子。

## 6. 总体数据流

分成"离线测量"和"运行时初始化"两段，这条分界是设计关键。

```
【离线，一次性，读源数据】scripts/build_marginals.py
   schema + queries + 源数据
   → 数值属性: 从查询提取切点造箱; 类别属性: 每值一箱
   → 在源数据上数每箱人数(这就是 1-way 测量, 现在噪声=0)
   → 写出 configs/<dataset>/init_marginals.json (含箱边界 + 计数)   ← 方案(b)

【运行时，不读源数据】run.py → run_evolution → generator
   只读 init_marginals.json
   → 每属性按计数确定性填配额 + 箱内均摊 + 每列独立打乱
   → S_0
```

### 6.1 与铁律 6 的关系

铁律 6 管的是"运行期不读私有答案"。测量（数每箱人数）与查询文件里的 `result`
字段一样，是**离线预计算**的合法 DP 接口，以后在这一步加噪。运行时只读 marginals
文件，从不碰源数据。build 脚本会带"读源数据"警告标注，与 `queries.load_data`
现有警告一致。

## 7. 实现计划

### 7.1 要改/新增的文件

| 文件 | 动作 | 内容 |
|------|------|------|
| `src/table_diffevo/marginals.py` | 新增 | `derive_bins_from_queries(queries, schema)`（纯函数，不读数据，可单测）、`load_marginals(path)`、`init_from_marginals(n_records, schema, marginals, rng)` |
| `scripts/build_marginals.py` | 新增 | 离线入口：读 schema+queries+源数据 → 造箱 → 数计数 → 写 `configs/<dataset>/init_marginals.json` |
| `src/table_diffevo/generator.py` | 改 | `init_synthetic_table` 加可选入参 `marginals=None`；为 None 时行为完全不变（向后兼容），非 None 时走 `init_from_marginals` |
| `src/table_diffevo/evolution.py` | 改 | `run_evolution` 加 `init_method='random'`（默认）与 `marginals=None` 两参；第 199 行按开关分派 |
| `scripts/run.py` | 改 | 加 `INIT_METHOD`、`MARGINALS_PATH` 常量，加载并透传 |
| `tests/test_marginals.py` | 新增 | 见 7.3 |

现有 `queries.py` / `fitness.py` / `sampling.py` / `distance.py` **一字不改**。

### 7.2 关键算法

**a) `derive_bins_from_queries`（数值属性）**
扫所有查询条件，按第 4.1 表收集切点；并入 schema 域端点排序去重 → 整数闭箱 `[lo,hi]`。
无查询覆盖的数值属性 → 不写进文件 → 运行时回退随机。白名单外算子 → 跳过该查询 + 提醒。

**b) 计数（build 脚本，读源数据）**
数值：每箱 `[lo,hi]` 数源数据落入记录数。类别：每合法值数记录数。

**c) `init_from_marginals`（运行时，每列独立）**
- 配额：值/箱 v 填 `count_v` 条。
- 凑 N：最大余数法调到正好 N（无噪时天然为 N；为将来加噪保留）。
- 箱内均摊（仅数值）：见第 5 节，确定性无随机。
- 每列独立打乱：填完是块状，须对每列用各自随机置换打散，否则各列同位置对齐、
  凭空造出属性间关联。用传入的 seeded rng，固定种子 → 可复现。
- 列间独立：各列分别处理，联合关系留给 3-way 演化。

### 7.3 测试（tests/test_marginals.py）

- 造箱：各白名单算子切点正确、对齐查询、域端点纳入、去重。
- 语义一致性：对每个白名单算子，验证"造出的箱边界 ⟺ 该查询在初始化表上精确命中"，
  把 `between` 端点、`<` vs `<=` 的差一位钉死。
- 边缘精确匹配：填出的表每箱人数 == 目标计数（无噪时精确）。
- 凑 N：构造带小数/不和为 N 的计数，验证最大余数法后正好 N。
- 每列被打乱且列间无人为关联（相关性接近 0）。
- 复现性：同 seed → 完全相同的 S_0。
- 回退：数值属性无查询覆盖 → 该列走随机；白名单外算子 → 跳过该查询、其余照常、
  评价仍正确；`marginals=None` → 与旧 `init_synthetic_table` 逐值相同（回归）。
- 端到端：build 产出文件喂给 run_evolution，初始 loss 显著低于随机初始化。
- 全套现有 176 passed 不回归。

## 8. 优缺点

**优点**
- 通用：箱由查询自动诱导，换数据/查询零改代码。
- 起点更好：1-way 一次到位，初始 loss 大降，演化专注高阶关系。
- 确定性：做法 B，无随机噪声（除每列打乱，已固定种子），可复现、可审计。
- 低风险：`marginals=None` 完全向后兼容；开关可一键切回随机做对照。
- DP 就绪：测量/加噪接口留在 build 脚本，运行时不碰源数据。
- 算子演化可预测降级：新算子最坏"帮不上忙"，不出错。

**缺点 / 代价**
- 多一份 `init_marginals.json` 要维护，**查询变了必须重新生成**（箱是查询诱导的）；
  build 脚本从当前查询重算，并在文件里记 queries 来源/版本以降低踩坑。
- build 脚本读源数据（离线合法，但要清楚标注，别混进运行时）。
- 箱内均摊是"公开假设"下的形状，不是真实箱内分布（现阶段无噪可接受；AIM 文档 7.5
  也承认这是额外假设）。

## 9. 需要留意 / 潜在遗漏

1. 初始化后 1-way 查询已满足，但演化仍可能扰动它们——这衔接后续"single 名额转
   3-way"的想法；本次不动查询集，只做初始化。
2. dtype：类别 nltcs 是 int 0/1、test_300 是字符串；数值填 int。复用现有
   `np.asarray` 口径，保证 dtype 正确、下游查询/距离不受影响。
3. 切点落在域外时裁剪到域内。
4. 诊断：`diagnostics.params` 加 `init_method`，以后看结果知道用了哪种初始化。
